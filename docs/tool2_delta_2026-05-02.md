Gap 3 is fully implemented in `eeat_scorer.py:EEATScorer` and `eeat_scorer.py:EEATScore`:

- `EEATScore` (eeat_scorer.py:22–38) — Dataclass capturing per-page EEAT record:

### Gap 4 — Internal Linking Detection

**What v2 Spec Required:**
Detect internal linking patterns; identify hub candidates; emit cluster signals.

**What Code Currently Does:**

Gap 4 is fully implemented in `cluster_detector.py:ClusterDetector` and `cluster_detector.py:ClusterResult`:

- `ClusterResult` (cluster_detector.py:16–35) — Per-domain link cluster record:
  - `domain`: Competitor domain
  - `pages_analyzed`: Count of pages examined (typically 3 per domain)
  - `internal_link_graph`: Dict representation of directed graph {source_url: [target_urls]}
  - `hub_candidates`: List of URLs with in-degree >= threshold (default 2)
  - `cluster_signal`: Enum (isolated, linked, clustered, insufficient_data)
  - `resolution_caveat`: Important note about scope limitations
- `ClusterDetector.analyze_domain(domain, pages)` (cluster_detector.py:41–120) — Graph analysis:
  - Builds directed graph of internal links across scraped pages
  - Identifies hubs (high in-degree >= threshold, default 2)
  - Emits signal: isolated (no internal links), linked (some links), clustered (strong hub structure), insufficient_data (too few pages)
  - Thresholds configurable via shared_config.json cluster_thresholds

**Configuration (shared_config.json):**
- `cluster_thresholds.hub_in_degree_threshold`: 2 (default)
- `cluster_thresholds.min_pages_for_signal`: 3 (min pages needed for confident signal)

**Important Caveat (cluster_detector.py:4–11):**
```
"Cluster detection runs only on the pages this audit scraped (typically 3 per
competitor). It cannot see the wider site structure. A domain marked 'isolated'
may still have a strong internal-link cluster invisible to this audit. Treat the
signal as suggestive, not decisive."
```

**What's Already Done:**
- ✅ ClusterResult dataclass with graph, hubs, cluster signal
- ✅ Directed graph construction from scraped pages
- ✅ Hub candidate identification (in-degree threshold)
- ✅ Cluster signal emission (isolated, linked, clustered, insufficient_data)
- ✅ Configuration-driven thresholds
- ✅ Important caveat about limited visibility into full site structure

## Bugs and Limitations Worth Flagging

### 1. EEAT Caveat (Not a Bug, But Important)

**File:** eeat_scorer.py

**Issue:** EEAT scores are heuristic proxies, not Google's actual EEAT model.

**Evidence:** eeat_scorer.py:1–8 states clearly: "These are heuristic proxies based on SEO industry conventions, NOT Google's actual EEAT model. Google does not publish its exact EEAT criteria."

**Impact:** Scores should not be presented to clients as authoritative Google EEAT. Position as structural competitive signals only.

---

### 2. Cluster Detection Scope Limitation (Not a Bug, Design Constraint)

**File:** cluster_detector.py:4–11

**Issue:** Internal linking analysis limited to pages scraped (typically 3 per domain). Cannot see full site structure.

**Evidence:** Resolution caveat explicitly documented: "A domain marked 'isolated' may still have a strong internal-link cluster invisible to this audit."

**Impact:** Cluster signal should be treated as suggestive, not definitive. Note that full internal-linking audit would require domain crawl or third-party API (Ahrefs, Moz).

**Suggestion for Spec Writer:** Consider recommending Ahrefs/Moz integration for future iteration if internal-linking strategy becomes core to competitive analysis.

---

### 3. Missing Error Handling in orchestrator.py

**File:** orchestrator.py

**Issue:** Streamlit UI does not gracefully handle subprocess failures beyond return code checking.

**Evidence:** orchestrator.py:60–65 checks `process.returncode` but does not provide recovery path (e.g., retry, alternative step).

**Impact:** If a step fails, user sees "Completed" message but result may be partial. Subsequent steps may depend on success of previous step.

**Suggestion for Spec Writer:** Propose step-level error handling with user prompts (e.g., "Step 2 failed. Retry? Skip? Abort?").

---

### 4. Backwards Compatibility: Legacy Tables

**File:** database.py

**Issue:** Multiple tables (competitor_metrics, semantic_audits) marked as legacy for migration phase.

**Evidence:** database.py:79–102 notes: "Legacy tables maintained for compatibility during migration"

**Impact:** Schema is carrying dead code. Clean migration path needed.

**Suggestion for Spec Writer:** Propose migration deprecation timeline (e.g., "Remove legacy tables in iteration N").

---

### 5. Orchestrator Hardcoded Step Behavior

**File:** orchestrator.py:47

**Issue:** Note in UI says "Step 3 includes Step 2 internally," but this logic is hardcoded in the Streamlit app, not in the underlying scripts.

**Evidence:** orchestrator.py:47 checks `step1` and `step2` checkboxes independently but note warns Step 3 subsumes Step 2.

**Impact:** Confusing UX. User might uncheck Step 2, thinking Step 3 will handle it, but unclear if both run or just Step 3.

**Suggestion for Spec Writer:** Propose step-level dependency specification (e.g., config-driven DAG) rather than UI-level notes.
